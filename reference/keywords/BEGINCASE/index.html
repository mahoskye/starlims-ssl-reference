    BEGINCASE¶

    Starts a :BEGINCASE block for evaluating one or more boolean :CASE conditions, with an optional :OTHERWISE branch and a required :ENDCASE.

    BEGINCASE opens SSL's case-block form. Each :CASE contains a full boolean expression, not a simple value label. The runtime evaluates the :CASE expressions in order. When a :CASE expression is true, that case body runs. If the body does not end with :EXITCASE;, execution continues and later :CASE expressions are still evaluated. An :OTHERWISE block runs only when no earlier :CASE body has executed.

    Behavior¶

    BEGINCASE must be followed by at least one :CASE block and closed with :ENDCASE;. Each :CASE expression is evaluated as a boolean condition.

    Without :EXITCASE;, a matched branch does not automatically stop the block. Later :CASE expressions are still checked, and additional matching case bodies may run. This is why :EXITCASE; is the normal pattern when you want the block to behave like an either-or decision structure.

    :OTHERWISE; is optional but recommended. It runs only if no earlier :CASE body executed. Once any :CASE body has run, :OTHERWISE; is skipped.

    When to use¶

    • When you need to evaluate several boolean conditions in a clearer structure than a long :IF chain.
    • When the logic benefits from named branches and an explicit default path.
    • When you intentionally want either single-branch behavior with :EXITCASE; or controlled multi-match behavior by omitting it.

    Syntax¶

    :BEGINCASE;
:CASE <condition>;
    /* statements;
    :EXITCASE;
:CASE <condition>;
    /* statements;
    :EXITCASE;
:OTHERWISE;
    /* statements;
    :EXITCASE;
:ENDCASE;

    BEGINCASE itself does not take parameters.

    Keyword group¶

    Group: Control Flow Role: opener

    Best practices¶

    Do

    • End each :CASE and :OTHERWISE branch with :EXITCASE; unless you intentionally want multi-match behavior.
    • Add an :OTHERWISE; branch when the logic needs a clear default path.
    • Use :BEGINCASE; when several boolean conditions are easier to read as branches than as a long :IF chain.

    Don't

    • Assume :BEGINCASE; stops after the first true branch. Without :EXITCASE;, later :CASE expressions are still evaluated and more than one body can run.
    • Omit :OTHERWISE; when the block needs explicit default handling.
    • Use :BEGINCASE; for trivial two-way branching where :IF and :ELSE would be clearer.

    Caveats¶

    • SSL keywords are case-sensitive. Write :BEGINCASE, :CASE, :OTHERWISE, :EXITCASE, and :ENDCASE in uppercase.

    Examples¶

    Condition-based notification¶

    Choose one notification path by ending each branch with :EXITCASE;. With nResultValue set to 85, the warning branch matches and the notification message is displayed.

    :PROCEDURE NotifyAnalysisStatus;
    :DECLARE sSampleID, sNotifyMessage, oEmail, nResultValue;

    sSampleID := "LAB-2024-0042";
    nResultValue := 85;
    oEmail := Email{};

    :BEGINCASE;
    :CASE nResultValue >= 100;
        sNotifyMessage := "CRITICAL: " + sSampleID
            + " exceeded the safe threshold";
        oEmail:Subject := "Critical Alert - Sample " + sSampleID;
        :EXITCASE;
    :CASE nResultValue >= 75;
        sNotifyMessage := "WARNING: " + sSampleID
            + " is approaching the threshold limit";
        oEmail:Subject := "Warning Alert - Sample " + sSampleID;
        :EXITCASE;
    :CASE nResultValue >= 50;
        sNotifyMessage := "NOTICE: " + sSampleID
            + " is within the acceptable range";
        oEmail:Subject := "Normal Status - Sample " + sSampleID;
        :EXITCASE;
    :OTHERWISE;
        sNotifyMessage := "INFO: " + sSampleID
            + " is below the typical range";
        oEmail:Subject := "Information - Sample " + sSampleID;
        :EXITCASE;
    :ENDCASE;

    oEmail:Body := sNotifyMessage;
    oEmail:To := {"labmanager@example.com"};

    InfoMes(sNotifyMessage);

    :RETURN sNotifyMessage;
:ENDPROC;

/* Usage;
DoProc("NotifyAnalysisStatus");

    InfoMes displays:

    WARNING: LAB-2024-0042 is approaching the threshold limit

    Dynamic pricing configuration¶

    Ordered conditions ensure the first matching business rule wins. With nBasePrice of 100, a contract order of 1500 units triggers the contract volume discount.

    :PROCEDURE CalculateDynamicPrice;
    :PARAMETERS nBasePrice, sCustomerType, nOrderVolume, bHasContract;
    :DEFAULT sCustomerType, "RETAIL";
    :DEFAULT nOrderVolume, 0;
    :DEFAULT bHasContract, .F.;
    :DECLARE nFinalPrice, sFormulaUsed, sMessage;

    nFinalPrice := nBasePrice;
    sFormulaUsed := "Base Price";

    :BEGINCASE;
    :CASE bHasContract .AND. nOrderVolume > 1000;
        nFinalPrice := nBasePrice * 0.70;
        sFormulaUsed := "Contract Volume Discount";
        :EXITCASE;
    :CASE sCustomerType == "WHOLESALE";
        nFinalPrice := nBasePrice * 0.80;
        sFormulaUsed := "Wholesale Rate";
        :EXITCASE;
    :CASE sCustomerType == "DISTRIBUTOR" .AND. nOrderVolume > 500;
        nFinalPrice := nBasePrice * 0.85;
        sFormulaUsed := "Distributor Volume Rate";
        :EXITCASE;
    :CASE nOrderVolume > 2000;
        nFinalPrice := nBasePrice * 0.90;
        sFormulaUsed := "High Volume Discount";
        :EXITCASE;
    :OTHERWISE;
        sFormulaUsed := "Standard Pricing";
        :EXITCASE;
    :ENDCASE;

    sMessage := "Price: " + LimsString(nBasePrice)
                + " -> " + LimsString(nFinalPrice);
    sMessage += " via " + sFormulaUsed;

    UsrMes(sMessage);

    :RETURN nFinalPrice;
:ENDPROC;

/* Usage;
DoProc("CalculateDynamicPrice", {100, "RETAIL", 1500, .T.});

    UsrMes displays:

    Price: 100 -> 70 via Contract Volume Discount

    Single-branch shipment processing¶

    Show the normal single-branch behavior by ending each branch with :EXITCASE;. Passing bRushOrder and bColdChain as .T. causes the first matching case body to execute.

    :PROCEDURE BuildShipmentActions;
    :PARAMETERS bRushOrder, bColdChain, bInternational;
    :DEFAULT bRushOrder, .F.;
    :DEFAULT bColdChain, .F.;
    :DEFAULT bInternational, .F.;
    :DECLARE aActions;

    aActions := {};

    :BEGINCASE;
    :CASE bRushOrder;
        AAdd(aActions, "Print priority label");
        :EXITCASE;
    :CASE bColdChain;
        AAdd(aActions, "Pack with temperature control");
        :EXITCASE;
    :CASE bInternational;
        AAdd(aActions, "Attach customs paperwork");
        :EXITCASE;
    :OTHERWISE;
        AAdd(aActions, "Standard packaging only");
        :EXITCASE;
    :ENDCASE;

    :RETURN aActions;
:ENDPROC;

/* Usage;
DoProc("BuildShipmentActions", {.T., .T., .F.});

    In this example, a shipment that is both rush and cold-chain adds only the priority-label action because the first matching branch exits the block. If any :CASE body runs, :OTHERWISE; is skipped.
